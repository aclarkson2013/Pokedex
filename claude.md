# Pokedex - Claude Project Guide

## Project Overview
A Progressive Web App (PWA) Pokedex with Pokemon search, game walkthroughs, team builder, and social features. Built with Next.js 15, TypeScript, Tailwind CSS 4, and Firebase.

## Tech Stack
- **Framework**: Next.js 15 (App Router, SSG)
- **Language**: TypeScript (strict mode)
- **Styling**: Tailwind CSS 4 (utility-first, no CSS modules)
- **State**: Zustand (client) + TanStack Query (server)
- **UI Primitives**: Radix UI
- **Auth/DB**: Firebase (Auth, Firestore, Cloud Functions)
- **Search**: Fuse.js (fuzzy matching)
- **Local Storage**: IndexedDB via `idb`
- **PWA**: Serwist (service worker)
- **MDX**: next-mdx-remote (walkthrough content)
- **Deployment**: Vercel

## Architecture
- Pokemon data fetched at **build time** from PokeAPI v2 REST → stored as static JSON in `/data/`
- IndexedDB caches data client-side for offline access
- 1025 Pokemon detail pages are statically generated (SSG)
- Walkthrough content is MDX files in `/content/walkthroughs/`
- User data (auth, collections, teams, friends) lives in Firebase Firestore
- Offline mutations queued in IndexedDB → synced on reconnect

## Directory Structure
- `src/app/` — Next.js App Router pages
- `src/components/` — React components (ui/, pokemon/, guides/, team/, social/, layout/)
- `src/lib/` — Core libraries (pokemon/, firebase/, db/, hooks/, utils/)
- `src/stores/` — Zustand stores
- `src/providers/` — React context providers
- `content/walkthroughs/` — MDX walkthrough files per game
- `scripts/` — Build-time scripts (data fetching)
- `data/` — Generated Pokemon data JSON (gitignored, regenerated at build)
- `public/` — Static assets (manifest, icons, splash screens)
- `firebase/` — Firestore rules, indexes, Cloud Functions

## Coding Conventions
- Use TypeScript strict mode everywhere
- Prefer `interface` over `type` for object shapes
- Use named exports (not default) for components and utilities
- File naming: kebab-case for files, PascalCase for components
- One component per file
- Collocate tests with source files (*.test.ts)
- Use `@/` path alias for imports from `src/`
- Tailwind for all styling — no inline styles, no CSS modules
- Server Components by default; add `"use client"` only when needed
- Use React Server Components for data fetching where possible
- Hooks in `src/lib/hooks/` prefixed with `use`

## Key Patterns
- **Data loading**: Static JSON → IndexedDB → React hooks → Components
- **Pokemon search**: Fuse.js against pre-built index (all local, instant)
- **Auth guard**: Wrap protected routes with `<AuthGuard>` component
- **Offline**: Service Worker (Serwist) + IndexedDB + mutation queue
- **Walkthroughs**: MDX files with custom components (`<PokemonLink>`, `<GymBattle>`, etc.)

## Common Commands
- `npm run dev` — Start dev server with Turbopack
- `npm run build` — Production build (runs SSG for all pages)
- `npm run fetch-data` — Fetch Pokemon data from PokeAPI (build-time script)
- `npm run lint` — ESLint

## Important Notes
- `/data/*.json` files are gitignored — they're regenerated by `fetch-data` script
- PokeAPI has no rate limit on REST but be respectful — the fetch script handles this
- All Pokemon IDs are 1-based integers (1 to 1025+)
- Type colors defined in `src/lib/pokemon/type-colors.ts`
- Firebase config stored in `.env.local` (gitignored)
